# Task List

Task List is a local-first CLI for managing tasks stored as Markdown files with YAML front matter.

Each task is a plain text file in a directory, making tasks easy to inspect, edit, back up, and version with Git.

## Goals

- store tasks as Markdown files
- use front matter for task metadata
- keep the format human-readable and manually editable
- provide a simple CLI for common task operations
- support machine-readable output with a global `--json` option

## Commands

- `list` — list tasks; for now it shows the id and title, and supports filtering by priority and whether a task is ready
- `add` — create a task; only `title` is required
- `view` — show a task with all metadata and the raw Markdown body; no rendered Markdown
- `update` — update any task field except `id`, including the Markdown body
- `delete` — permanently remove a task
- `dep` — manage task dependencies

A global `--json` option should output command results as JSON.

A global `-d, --directory` option should select the directory containing task Markdown files. `TASKLIST_DIRECTORY` is also supported. See [Directory Selection](#directory-selection) for resolution order and configuration details.

## Task Format

Each task is a Markdown file with YAML front matter and an optional Markdown body.

Example:

```md
---
id: task-7k9m
title: Buy groceries
status: todo
priority: 2
parentId: task-3h7q
dependsOn:
  - task-2w8x
createdAt: 2026-03-18T10:00:00Z
lastModified: 2026-03-18T10:00:00Z
---

Need milk, eggs, and bread.
```

Front matter fields:

- `id` — unique task identifier; not updatable
- `title`
- `status`
- `priority`
- `parentId` — optional parent task id
- `dependsOn` — optional list of task ids this task depends on
- `createdAt` — set automatically by the CLI by default; user may override
- `lastModified` — set automatically by the CLI by default; user may override

Computed field:

- `ready` — true when all dependencies are completed; false when one or more dependencies are not yet completed

The Markdown body is also part of the task and is updatable.

## ID Generation

Task IDs should be auto-generated by the CLI in this format:

- `<prefix><hash>`

Where:

- `<prefix>` comes from the tasklist directory config file `.tasks`
- the config file is a flat `key=value` file
- the supported key is `prefix`
- the default prefix is `task-`
- `<hash>` is lowercase Crockford Base32
- `<hash>` is exactly 4 characters
- `<hash>` is derived from a deterministic FNV-1a 64-bit hash

Lowercase rendered alphabet:

- `0123456789abcdefghjkmnpqrstvwxyz`

Excluded characters:

- `i`, `l`, `o`, `u`

The CLI should:

1. build a seed from `title|description|creator|createdAtUnixNano|nonce`
2. hash it with Go `fnv.New64a()`
3. encode the `uint64` hash using Crockford Base32
4. left-pad with `0` if needed
5. keep the last 4 characters
6. render the suffix in lowercase
7. prefix with the configured `prefix` value from `.tasks`
8. if no config is present, use the default prefix `task-`
9. retry with incrementing nonce values on collision

This gives `32^4 = 1,048,576` possible visible IDs.

See `docs/id-generation.md` for the full spec and Go reference implementation.

## Status Values

- `todo`
- `in_progress`
- `done`

If status is omitted, it should default to `todo`.

## Priority Values

- `0` — lowest priority
- `1`
- `2`
- `3` — most urgent

If priority is omitted, it should default to `0`.

## Task Relationships

A task may have:

- one optional parent task
- many child tasks

This parent-child relationship is for grouping tasks. It is separate from dependencies.

Dependencies should be stored in `dependsOn` as a list of task ids.

A task also has a computed `ready` field. `ready` is true only when all dependencies are completed.

The CLI will not prevent circular dependencies automatically, but it should provide commands to inspect and report them.

If a parent task has child tasks, deleting the parent should emit a warning and fail by default. The parent cannot be deleted until its child tasks are moved outside of that parent, unless deletion is invoked with `-f` or `--force`. With the force option, deleting the parent may proceed and should also delete all of its child tasks.

## Dependency Commands

The CLI should provide a `dep` command with these subcommands:

- `dep add <task> <depends-on>`
  - Add a dependency
  - Meaning: `<task>` depends on `<depends-on>`

- `dep remove <task> <depends-on>`
  - Remove a dependency
  - Alias: `dep rm`

- `dep list <task>`
  - List dependencies for a task
  - Supports directions:
    - `--direction down` = what this task depends on
    - `--direction up` = what depends on this task
    - `--direction both`

- `dep tree <task>`
  - Show a dependency tree rooted at the task
  - Options include:
    - `--direction`
    - `--max-depth`
    - `--format text|mermaid`

- `dep cycles`
  - Detect and report dependency cycles

## File Naming

Task files should be named:

- `<id>.md`

## Principles

- local-first
- plain text
- scriptable
- git-friendly
- easy to edit manually

## Directory Selection

The CLI should read and write task Markdown files from a task directory determined as follows:

1. `-d, --directory` global CLI option
2. `TASKLIST_DIRECTORY` environment variable
3. default `./tasks`

The task directory may also contain a `.tasks` configuration file.

The `.tasks` file is a flat `key=value` file and should support:

- `prefix` — the prefix used when generating task IDs

If `prefix` is not set, the default prefix is `task-`.

## Open Issues

1. Seed uses "description" but the task format calls it "body". The seed format references `title|description|creator|...` but there is no "description" field — the task has a Markdown body. This should be consistent.
2. Seed references "creator" but there is no creator field. There is no `creator` field defined in the task format, so it's unclear where this value comes from.
3. `delete` behavior with dependencies is unspecified. If you delete a task that other tasks depend on (via `dependsOn`), what happens to those references? Only parent-child deletion is addressed.
4. `update` should auto-update `lastModified`. It's not stated that `lastModified` should be automatically updated when a task is changed via `update`.
5. `ready` definition of "completed" is ambiguous. It says "all dependencies are completed" but doesn't say what "completed" means. Presumably `status: done`, but this should be explicit.
