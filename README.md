# Todo List

Todo List is a local-first CLI for managing todos stored as Markdown files with YAML front matter.

Each todo is a plain text file in a directory, making todos easy to inspect, edit, back up, and version with Git.

## Goals

- store todos as Markdown files
- use front matter for todo metadata
- keep the format human-readable and manually editable
- provide a simple CLI for common todo operations
- support machine-readable output with a global `--json` option

## Planning

Planning documents, user stories, and supporting specifications are stored in the repository todo directory as todos.

Use the CLI to inspect them:

```bash
todolist list
todolist view <todo-id>
```

## Releases

Pushing a semantic version tag triggers the GitHub Actions release workflow.

Example:

```bash
git tag v1.2.3
git push origin v1.2.3
```

The workflow builds release archives for macOS, Linux, and Windows on amd64 and arm64, generates a `SHA256SUMS` file, and creates the matching GitHub Release.

## Shared Specifications

### ID Generation

Todo IDs should be auto-generated by the CLI in this format:

- `<prefix><counter>`

Where:

- `<prefix>` comes from the todolist directory config file `.todos`
- the config file is a flat `key=value` file
- supported keys are `prefix` and `last_id`
- the default prefix is `todo-`
- `<counter>` uses lowercase Crockford Base32
- `<counter>` is exactly 4 characters

Lowercase rendered alphabet:

- `0123456789abcdefghjkmnpqrstvwxyz`

Excluded characters:

- `i`, `l`, `o`, `u`

The CLI should:

1. read `last_id` from `.todos`
2. if `last_id` is missing, start at `0000`
3. otherwise decode the `last_id` suffix and increment by 1
4. render the 4-character suffix in lowercase
5. prefix with the configured `prefix` value from `.todos`
6. if the candidate ID already exists, increment again and retry
7. on success, write the new ID back to `last_id`
8. if all `0000`..`zzzz` IDs are exhausted, fail with an ID-space-exhausted error

This gives `32^4 = 1,048,576` possible visible IDs.

### Directory Selection

The CLI reads and writes todo Markdown files from `./todo`.

```bash
todolist list
```

The todo directory may also contain a `.todos` configuration file.

The `.todos` file is a flat `key=value` file and should support:

- `prefix` â€” the prefix used when generating todo IDs
- `last_id` â€” the most recently allocated todo ID for incremental ID generation

If `prefix` is not set, the default prefix is `todo-`.
